Graphics in a Webpage

an introductory tutorial

main ways to get graphics:

  • Images
  • SVG
  • HTML Canvas

We'll be focusing on the last two bullet points. The first way, images, is pretty self-explanatory and is taught in KA's HTML and CSS course.

Note that you can technically use HTML elements like with CSS styling to create graphics. However, that requires a lot more work than it's worth and can only create simplistic shapes at best. Both SVG and the HTML canvas would give you many more options with much more ease.

SVG

SVG stands for Scalable Vector Graphics and can be used to create vector-based graphics in webpages. It is basically a markup language, just like HTML, except instead of elements like div or p, we have graphics elements like line or circle.

Typically, for more complex images, people will use graphics editors like Inkscape to create their SVGs for them instead of hand-coding things. If you don't have the option to do so, you can always code it yourself with relative ease.

Inline SVGs can easily be added to your HTML like so:

  • Resolution independent and very flexible--the edges will remain sharp even when stretched and scaled because they're vectors
  • Elements can be accessed in the DOM and have the default built-in event handlers
  • Elements can be easily manipulated and styled with CSS and JS
  • Is better than canvas at rendering a small amount of large objects
  • Becomes very slow when rendering a large number of objects
  • Not very good with complex animations and game graphics

Canvas

The HTML canvas is a JS-based way of creating graphics for webpages. It is somewhat similar to the PJS environment that KA provides--in fact, KA automatically loads the Processing JS library and the HTML canvas for you behind the scenes whenever you start a new PJS program.

You don't necessarily need to use the Processing JS library to use the HTML canvas, though, since everything you need is built in. Be warned that using the canvas does take a little more effort to initially set up than SVG.

  • Easy pixel manipulation

  • Elements in your graphic can't be accessed in the DOM

How do you determine how your SVG graphic will scale to fit the element's dimensions? Well, that's where the magical attribute called viewBox comes in.

Those four numbers determine the coordinate system used by your SVG elements to position themselves. When you create an element with an x value of, say, "10," then that "10" is interpreted in terms of the coordinate system viewBox sets up.

The first two numbers, 0 and 0 in the example above, determine the minimum x and y values that can be seen in your graphic. It basically determines where the point (0, 0) in your coordinate system will appear in terms of the top left corner of the SVG element. Like in PJS, the larger the x value, the further right the origin is, and the larger the y value, the further down the origin is.

The last two numbers determine the width and the height of the coordinate system, respectively, which in turn determines how the graphic will be scaled to fit the SVG element's width and height.

You can also change the styling of the stroke easily, with a wide range of attributes you can set:

  • stroke-width is a number that sets the thickness of the stroke.
  • stroke-linecap can be set to butt, round, or square and controls the appearance of the line's end, much like what the strokeCap function does in PJS.
  • stroke-linejoin can be set to arcs, bevel, miter, miter-clip, or round and control the appearance of corners where two lines meet. This is similar to what the strokeJoin function does in PJS. For a visualization of what it does, visit this link (leads off KA).

For example, setting it to "4" would mean each dash and each gap would be 4 units long, whereas setting it to "4 1" means that each dash is 4 units long and each gap is 1 unit long. "4 1 2" will make it so the first dash is 4 units, the first gap is 1 unit, the second dash is 2 units, the second gap is 4 units, the third dash is 1 unit, and so on. You can look at this link (leads off KA) for a visualization and more information on this attribute.

Now you can create straight-edged, irregular shapes, but what about those with curved edges?

That's where path comes in! path can create pretty much any shape you want. It has a d attribute, a string that contains all the path commands that define the path you want to create. Path commands start with a letter representing the type of command followed by numbers that are the coordinates being passed into the command.

These commands are case sensitive--capital letters mean that the numbers represent an absolute position, whereas lowercase letters mean that the numbers represent how much you want to change from the current position. The commands are as follows:

  • M and m are moveTo commands that take two parameters (new x and y or change in x and y). Think of it as picking up the pen that you're using to draw the path and moving it to a new, different coordinate. No mark will be made connecting the current point to the point you want to move to.
  • Z or z automatically closes the path, drawing a straight line between the last point and the first point (if they are different).
  • L and l represent lineTo commands that take two parameters (new x and y or change in x and y). It draws a straight line from your current point to the point you specify.
  • H and h are also lineTo commands that take one parameter (the new x or change in x). They draw a horizontal line from the current coordinate to whatever new x you specify. Very similar are the V and v commands, which draw a vertical line from the current point to a new y value.
  • C and c are cubic bezier curve commands. They take 6 parameters that represent either the new x and y coordinates or the change in x and y coordinates (with respect to the current position) for the first control point (c1), the second control point (c2), and the end point, in that order. If you are familiar with complex shapes in PJS, the C command will be the same as PJS's bezierVertex().
  • Q and q are quadratic bezier curve commands that take 4 parameters representing either the new x and y coordinates or the change in x and y coordinates for the control point and the end point (in that order).
  • A and a are arcTo commands that take 7 parameters and draw an elliptical arc:
    • The first two numbers specify the x radius and the y radius of the ellipse.
    • The third number specifies the number of degrees that the ellipse will be rotated.
    • The fourth number is either a 0 or a 1, with 0 meaning that the browser should draw the smaller arc and 1 meaning the browser should draw the larger arc.
    • The fifth number also is either a 0 or a 1, with 0 meaning the browser should draw the anticlockwise turning arc and 1 meaning the browser should draw the clockwise turning arc.
    • The last two numbers specify the end point of the arc.

The defs element in SVG is one of the most useful elements out there. It's short for definitions, and is where special elements like gradients or filters or just commonly used graphics will go to be stored for future use. Although elements can technically be referenced from anywhere, having a defs element helps consolidate those referenced elements and makes for more accessible code.

Anything you put in defs will not show up until you reference it later. You usually will need an id for the element so that you can easily access it later; it helps to name it something descriptive, like "skyGradient" or something of the sort, to prevent accidental duplication.

Using the canvas API to draw basic shapes might be pretty different from what you're used to. Many of the functions in PJS like line or triangle or arc have disappeared, and in their place is a lot of beginPath and closePath instead.

I'd recommend that you create your own functions for line and the like as you learn how to make the canvas API draw those shapes. This will make it easier and more familiar for you, and you'll be able to have your own basic version of PJS working in no time.

Rectangles

The easiest shape to draw with the canvas API is the rect element. There are three ways to draw rectangles immediately on to the canvas:

  • clearRect(x, y, width, height) is only visible if you have other things drawn on the canvas. It turns everything in the rectangle to transparent black and effectively erases everything in that region.
  • fillRect(x, y, width, height) draws a filled rectangle with the top left corner at (x, y) and with its width and height specified by the third and fourth parameters. You'll need to set the canvas's fillStyle to some color, like "green," in order to see thsi rectangle.
  • strokeRect(x, y, width, height) draws a stroked rectangle at (x, y) with the specified width and height. It has no fill and its stroke is controlled by the canvas's strokeStyle.

From here on out, drawing shapes on the canvas won't be so familiar and straightforward. I'll cover how to draw some other basic shapes like ellipses and lines in the next article, where we'll need to start using beginPath() and closePath().

Most of what we'll draw on the canvas will need beginPath() and closePath(). So what do these two things do?

beginPath() and closePath() start a new path and closes the path (attempts to draw a line between the current point and the first point in the current path) in the canvas, respectively. You define the boundaries and the shape of the path with the commands you put in between. If it helps you to compare this with PJS, the library's beginShape() and endShape(CLOSE) are equivalent to beginPath() and closePath().

Unlike PJS's shapes created with beginShape() and endShape(), however, to make a path show up, you'll need to actually call ctx.stroke(); or ctx.fill(); to stroke or fill your path. These two commands will affect only your current path, so be sure to call them before you start a new path if you don't want them to affect other paths.

Now that we've gotten some basic understanding of beginPath() and closePath(), let's actually use them to make some shapes!

Arcs

Unlike arcs in PJS, arcs with the built-in method will always be circular. They also won't be the pie-shaped slices that you're used to in PJS and will instead take on a shape similar to incomplete ellipses (appearing as if one part of the shape was loppped off in a line connecting the start and end points of the arc).

arc takes 5 parameters: the x, y, radius, startAngle, and endAngle. The x and y paramters control the coordinates of the center fo the arc, and the radius naturally controls the radius of the arc. The startAngle and endAngle are like the PJS arc's start and end angle parameters, except they're in radians.

To make pie-shaped arcs like you're used to in PJS, simply add a moveTo(x, y) (with the x and y being the center of the arc) before you call the arc.

Beziers

Those of you familiar with making graphics in PJS may be familiar with bezierVertex and its power when it comes to creating any curved, irregular shape. The good news is that the Canvas API's bezierCurveTo method is equivalent to bezierVertex and takes the same parameters: cx1, cy1, cx3, cy2, x2, and y2, which determine the coordinates of the two control points and the ending point of the bezier curve.

It's also worth noting that the Canvas API also offers another way of creating curves, through the quadraticCurveTo method. Unlike bezierCurveTo, quadraticCurveTo creates a quadratic bezier curve, not a cubic bezier curve, meaning that it only has one control point and not two. As a result of that, it takes 4 parameters, not 6: cx, cy, x2, and y2, which determine the coordinates of the one control point and the ending point, respectively.

This won't be a tutorial on what beziers are and how they function, since I'm assuming background knowledge in that area for PJS. If you're new to complex shapes, I'd recommend you check out KA's documentation on the PJS functions and play around with them a bit to gain a better understanding of them.

More arcs

Other than the arc method, the canvas also has an arcTo method that also draws a circular arc, but instead of specifying angles, you specify points instead.

The arcTo method takes 5 parameters: cx1, cy1, cx2, cy2, and r, which represent the coordinates of the two control points and the radius of the circular arc.

How do the control points affect the arc? Well, first imagine there are two line segments: one drawn from the current position to the first control point, and another drawn from the first control point to the second control point. You'll need arcTo if you want to create a canvas equivalent of the PJS rect function that is capable of creating rounded corners. Be aware that for very large radii, the arc may not start at the current point but instead go beyond it because it cannot fit in the area otherwise.

You've already seen, so far, how to change basic styling of elements by setting fillStyle and strokeStyle to different colors. In addition to using rgba for transparent colors, you can also set a globalAlpha property that is used to determine the opacity of all colors after you set it (with 0 being transparent and 1 being opaque):

Like in SVG and in PJS, there are also different properties you can set to change the styling of the stroke:

  • lineWidth is a number that determines the width of the stroke, equivalent to what the strokeWeight function does in PJS. The default value is 1.
  • lineCap can be set to "butt", "round", or "square" and controls the appearance of the end of lines. This is equivalent to PJS's strokeCap function, with butt equivalent to PJS SQUARE, round equal to PJS ROUND, and square equal to PJS PROJECT. The default value is butt.
  • lineJoin can be set to "round", "bevel", or "miter" and controls the appearance of the corners where two lines are joined. This is the same as what strokeJoin does in PJS. The default value is "miter"

What would a canvas be if it didn't have transformations? Many of your PJS transformations also easily translate to the Canvas API:

  • translate(x, y) works pretty much exactly the same way as before.
  • rotate(angle) also works similarly, except the angle (as always) is in radians.
  • scale(xAmt, yAmt) works the same, except you must specify both parameters. In PJS you could omit one parameter and the yAmt was automatically assumed to be the same, but in the canvas API that doesn't happen anymore.

There's also a transform function and setTransform function that combines many transformations into one handy function that manipulates the transformation matrix. Learn more about them with MDN's helpful reference (link leads off KA--setTransform is linked from the transform method's reference).

You might be asking whether we have any equivalent to pushMatrix, popMatrix, and resetMatrix in the Canvas API. As a matter of fact, we do!

pushMatrix's and popMatrix's equivalent methods are save and restore, respectively. A more accurate comparison actually can be made between pushStyle and popStyle and save and restore, since save saves not only the current transformation matrix but the current styling properties, too. restore, being aptly named, restores the canvas to its last saved state, so any styling changes or transformations you apply between save and restore won't affect other parts of your graphic.

resetMatrix's equivalent is resetTransform, which resets the canvas's transformations to its original state. This tends to be a less popular option (at least from what I've seen) for controlling transformations because it doesn't have the level of control that save and restore give you over which areas are affected by what transformations.

Processing JS is a pretty useful library built on the Canvas API and is probably what a lot of you are most familiar with. To use PJS inside a webpage, you can check out KA's article on using PJS in webpages to learn more.

The simplistic rundown of things is that you have the canvas element you want to draw on (as always), load in the PJS library, and then create a new Processing instance using the canvas and a function containing your graphics code as parameters. The function for drawing on the canvas will take one parameter, which is the drawing context, and all commands that refer to the canvas as the object need to be prefixed with this context.

However, note that Processing JS is no longer being updated (the last version is 1.6.6 and any further updates were discontinued in December of 2018). If you are doing things off KA, you should probably look into other libraries built on the canvas, such as p5js.

(KA even lists some other popular graphics libraries in their JS libraries article that you could look into. p5js is probably the most similar in syntax and functionality to PJS, though, and requires minimal tweaking)

Additionally, it's worth noting that the template KA gives you in the article above uses with, which according to MDN (link leads off KA) is not recommended and has been deprecated. Instead, when using the template, you'll want to prefix all commands with the processing instance like you need to do with the regular API (you may want to rename it to something shorter, like ctx). with(processingInstance) does the same thing (and with less effort, presumably) but it's never good to continue using deprecated commands if you have alternatives to it--they're deprecated for a reason, after all.

Whether you use PJS or other libraries or decide to stick with the built-in API, happy coding with the canvas!

Gradients

Like SVG, the Canvas API actually has a built-in way to create both linear and radial gradients very easily (in contrast to how you have to use for loops and lerpColor or translucency to create gradients in PJS).

Also like SVG, you will have to add in color stops specifying the colors you want in the gradient at some particular point (from 0 to 1) in the gradient. You'll do this with an .addColorStop() method after defining the position of the gradient, however. To apply it to a shape, set the fillStyle or strokeStyle to the variable that you assigned to the gradient.

MDN has helpful explanations on both linear gradients and radial gradients that you can check out for more information.

Shadows

The HTML canvas also provides various properties you can set to create shadows around drawn elements. The default setting is, of course, that there is no shadow, but once you set the properties so that the shadow shows up, all the elements will have shadows.

To set the amount of blur you want the shadow to have, use the shadowBlur property and set it to some nonzero number. The larger the number, the more blurry you get.

You can also change the shadow color with the shadowColor property, by setting it similarly to how you'd set the fill or stroke color.

Lastly, to control the positioning of the shadow with respect to the elements' positions, set the shadowOffsetX and shadowOffsetY properties that represent how far to the right and how far down the shadows will be shifted (negative numbers will shift the shadow to the left and up).

Patterns

Also like in SVG, you can create repeating patterns to fill an element (or to use as the stroke of an element) with the Canvas API.

A lot of different forms of images can be used as the image for the pattern to be repeated. MDN has a wonderful documentation of the createPattern method and how it can be used to create patterns from different images.

Clipping and Compositing

The Canvas API has many ways you can combine two images that you can set using the globalCompositeOperation property. There are 12 different operations you can perform, including things like multiply, overlay, darken, color dodge, and more, each with different effects.

You can also easily create clipping paths to only show parts of a graphic within a path using the clip method (which turns the current path into a path that clips whatever follows it).

For an introduction to clipping and to the compositing operations available, visit MDN's helpful page on the two topics. For more on clipping, visit MDN's documentation of clip as well.

imageData

Being able to manipulate individual pixels is one of the canvas's advantages. You can get image data using the getImageData method, with the added benefit of being able to specify the rectangular area from which you want to get pixel data. To update the pixels after modification, use putImageData with the added benefit of being able to determine where the top left corner of the painted pixel image will be.

For more information about imageData and its uses, visit MDN's tutorial on pixel manipulation.
